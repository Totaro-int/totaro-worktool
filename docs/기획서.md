## 1. 개요

### 1.1 배경과 문제

Totaro(totaro.co.kr)는 3인 규모 회사로, 세 갈래의 업무를 동시에 진행한다.

| 사업영역         | 내용                                | 관련 서비스                        |
| ---------------- | ----------------------------------- | ---------------------------------- |
| AI 브랜딩        | 스토어 관리 및 온라인 브랜딩        | totaro.co.kr/branding              |
| B2B 소싱 AI      | B2B 거래 소싱                       | cos.totaro.co.kr, web.totaro.co.kr |
| AI 에이전트 판매 | 마케팅 에이전트 등 AI 에이전트 판매 | —                                  |

문제는 업무가 **여러 채널에 흩어져 있다**는 점이다. 개발은 GitHub·Claude Code,
스토어 운영은 네이버, 마케팅·브랜딩은 네이버/문서로 진행되다 보니 "지금 누가
무슨 일을 했는지" 한눈에 보이지 않는다. 3인 팀이라 별도 보고 체계를 두기도
부담스럽다.

### 1.2 목표

- 흩어진 모든 업무를 **한 곳(통합 활동 피드)** 에 자동으로 모은다.
- "누가 / 어떤 영역에서 / 무슨 일을 했는지" 를 즉시 확인할 수 있다.
- 개발 업무는 GitHub에서, 스토어 업무는 네이버 API에서 **자동 수집**한다.
- 3인 팀이 입력 부담 없이 쓸 만큼 단순하게 유지한다.

### 1.3 한 줄 정의

> 토타로 3인 팀의 모든 업무(개발·네이버 스토어·마케팅·문서)를 한 곳에 모아
> "누가 무슨 일을 했는지" 통합 피드로 보여주는 사내 웹 대시보드.

### 1.4 사용자

- 사용자는 Totaro 팀원 3명뿐인 **내부 전용 도구**다.
- 외부 고객·게스트 접근은 없다.
- 권한은 단순하다 — 팀원은 모두 동등하게 모든 데이터를 보고 쓴다.
- (미해결) 3명의 역할 분담: 각자 한 영역을 전담하는지, 함께 보는지 → §12 참고.

## 2. 제품 원칙

1. **통합 활동 피드가 심장이다.** 모든 기능은 결국 "활동(activity)" 한 줄을
   만들어 피드에 쌓는다. 화면·연동이 늘어도 이 중심은 바뀌지 않는다.
2. **자동화 우선, 수동은 보조.** 수집할 수 있는 건 자동 수집한다. 자동화가
   불가능한 영역(네이버 블로그 등)만 수동 입력 폼으로 보완한다.
3. **3인 팀 규모를 넘지 않는다.** 복잡한 권한·승인·조직 구조를 만들지 않는다.
   기능은 "정말 쓰는 것"만 남긴다.
4. **단계적으로 키운다.** 자동화가 꽂힐 그릇(1단계)을 먼저 완성하고 연동을
   하나씩 붙인다. 각 단계는 그 자체로 완결적이어야 한다.
5. **비용은 무료 티어에서 시작한다.** 규모가 커질 때만 유료로 전환한다.

## 3. 전체 범위와 단계

| 단계 | 이름               | 범위                                          | 상태              |
| ---- | ------------------ | --------------------------------------------- | ----------------- |
| 1    | 핵심 대시보드      | 구글 로그인, 통합 활동 피드, 할 일 보드, 문서 | 프로토타입 구현됨 |
| 2    | GitHub 연동        | 커밋·PR 자동 피드, Claude Code 작업 감지      | 기획              |
| 3    | 네이버 커머스 연동 | 스마트스토어 주문·상품 변경 자동 수집         | 기획              |
| 4    | 확장               | 네이버 광고, 알림, 주간 업무 요약             | 기획              |

> 프로토타입을 기획에 맞춰 검토·정리하는 것을 1단계 개발의 출발점으로 삼는다.

## 4. 핵심 개념 — 통합 활동 피드

모든 업무는 출처가 무엇이든 `activities` 한 줄로 표준화되어 피드에 쌓인다.

```
[직접 기록]    ─┐
[할 일 완료]   ─┤
[문서 업로드]  ─┼─→  ╔═══════════════════╗  ─→  [활동 피드 화면]
[GitHub 커밋]  ─┤     ║  activities 테이블 ║       멤버·영역·출처·기간 필터
[네이버 이벤트] ─┘     ╚═══════════════════╝
```

활동 한 줄의 표준 형태:

| 속성                | 의미                                      |
| ------------------- | ----------------------------------------- |
| 담당자(member)      | 누가 한 일인가                            |
| 사업영역(work_area) | 어느 영역의 일인가                        |
| 출처(source)        | manual / task / document / github / naver |
| 제목·설명           | 무슨 일인가                               |
| 링크(url)           | 관련 외부 링크 (커밋, 주문 등)            |
| metadata            | 출처별 부가 데이터 (JSON)                 |
| 발생 시각           | 언제 일어났는가                           |

이 표준화 덕분에 새 연동이 늘어도 화면(피드)은 그대로 재사용된다.

## 5. 사용자 시나리오

- **아침 현황 파악** — 출근 후 활동 피드를 열어 어제~오늘 팀 전체가 무슨 일을
  했는지 1분 안에 확인한다.
- **영역별 점검** — "B2B 소싱 AI" 필터를 눌러 해당 영역의 진행 상황만 본다.
- **개인 업무 회고** — 본인 필터를 눌러 한 주 동안 한 일을 모아 본다.
- **개발 추적** — 누가 GitHub에 무엇을 커밋·머지했는지 피드에서 바로 본다.
- **스토어 모니터링** — 네이버 스토어 신규 주문·상품 변경이 자동으로 피드에 뜬다.
- **할 일 관리** — 사업영역별 할 일을 만들고 완료하면 자동으로 피드에 기록된다.
- **서류 공유** — 견적서·기획안 등을 올리면 팀이 보고 피드에도 남는다.

### 6.2 Phase 2~4 확장

- **members 에 `github_login` 추가** (Phase 2) — GitHub 작성자 ↔ 팀원 매핑.
- **github_repos** (Phase 2) — 연동 저장소 목록. `repo_full_name`,
  `work_area_id`(영역 매핑), `active`.
- **integration_state** (Phase 3) — 네이버 폴링 커서(마지막 동기화 시각/ID)
  같은 동기화 상태 저장.
- 네이버·GitHub 인증정보는 **DB가 아니라 환경변수/시크릿**으로 관리한다(보안).
- GitHub·네이버 이벤트는 모두 `activities` 로 들어간다 — 새 테이블 없이
  `source` 와 `metadata` 로 구분.

## 7. 화면 상세 (Phase 1)

### 7.1 로그인 (`/login`)

- 구글 계정 로그인 버튼 하나. 비밀번호 없음.
- **접근 제한**: 허가된 팀원 이메일만 로그인 가능해야 한다. 현재 프로토타입은
  구글 계정이 있으면 누구나 로그인된다 → **이메일 화이트리스트** 도입 필요
  (§11, §12).
- Supabase 미설정 시 "초기 설정 안내" 화면을 보여준다.

### 7.2 활동 피드 (`/dashboard`)

핵심 화면. 구성:

- **업무 직접 기록** (접이식 폼) — 제목, 상세, 사업영역, 업무 일시.
- **필터** — 멤버 / 사업영역 / 출처 칩. (추가 예정: 기간 필터)
- **피드 목록** — 최신순. 각 항목: 출처 배지, 제목, 설명, 외부 링크, 담당자,
  영역 배지, 상대 시각("3시간 전").
- (개선안) 날짜 구간 그룹핑 — "오늘 / 어제 / 이번 주" 헤더로 묶기.
- (개선안) 100건 초과 시 더 보기 / 페이지네이션.

### 7.3 할 일 보드 (`/tasks`)

- **3컬럼 칸반** — 할 일 / 진행 중 / 완료. 컬럼별 개수 표시.
- **새 할 일** (접이식 폼) — 제목, 상세, 사업영역, 담당자, 마감일.
- **카드** — 제목, 설명, 영역 배지, 담당자, 마감일. 상태 이동 버튼, 삭제 버튼.
- **완료 시 활동 피드에 자동 기록** (source=task).
- (개선안) 마감 임박/초과 카드 강조 표시.

### 7.4 문서 (`/documents`)

- **업로드** (접이식 폼) — 파일, 사업영역, 설명.
- **목록** — 파일명(다운로드 링크), 영역, 업로더, 크기, 날짜, 삭제.
- 업로드 시 활동 피드에 자동 기록 (source=document).
- 비공개 저장 — 다운로드는 서명 URL(만료 1시간)로 제공.
- (개선안) 사업영역별 보기, 파일명 검색.

### 7.5 공통

- **사이드바** — 로고, 메뉴(활동 피드 / 할 일 / 문서), 현재 사용자, 로그아웃.
- 라이트 테마, 슬레이트 + 인디고 색상. 사업영역은 고유색으로 구분.

## 8. 자동화 연동 설계

### 8.1 GitHub 연동 (Phase 2)

- **방식**: GitHub App(권장) 또는 저장소별 Webhook. 여러 저장소(cos·web·툴
  자체 등)를 다루므로 조직(Organization)에 GitHub App 설치를 권장. 간단히
  시작하려면 저장소별 Webhook + 공유 시크릿도 가능.
- **수집 이벤트**: `push`(커밋), `pull_request`(생성·머지). 필요 시 `issues`.
- **데이터 흐름**: GitHub → Webhook → `app/api/webhooks/github` → 서명 검증 →
  `activities` 삽입(source=github, metadata에 커밋 SHA·PR 번호·작성자).
- **Claude Code 작업 감지**: 커밋 메시지의 `Co-Authored-By: Claude` 트레일러를
  감지해 "Claude 작업" 배지 부여 (metadata 플래그).
- **매핑**: GitHub 작성자 → `members.github_login`, 저장소 → 사업영역.
- 실시간성: Webhook은 즉시 반영("바로바로" 요구 충족).

### 8.2 네이버 커머스 연동 (Phase 3)

- **방식**: 네이버 커머스 API. 판매자가 **커머스 API 센터에서 애플리케이션을
  등록**해 client id/secret 발급(전제 조건). 실시간 푸시가 약하므로 **주기적
  폴링** 방식.
- **스케줄러**: Vercel Cron은 무료(Hobby) 플랜에서 빈도 제한이 있다. 대안 —
  GitHub Actions 스케줄 워크플로우(무료, 분 단위) 또는 Supabase 예약 함수.
  비용 0원 유지를 위해 GitHub Actions cron 권장.
- **수집 데이터**: 신규 주문, 상품 등록·수정. (확장: 문의/리뷰)
- **데이터 흐름**: Cron → 네이버 API(마지막 커서 이후 조회) → `activities`
  삽입(source=naver) + `integration_state` 커서 갱신.
- **귀속**: 스토어 업무는 "AI 브랜딩" 영역으로, 담당자는 설정에서 지정한
  스토어 담당 팀원으로.

### 8.3 확장 연동 (Phase 4)

- **네이버 검색광고 API** — 광고 캠페인 성과를 피드/대시보드에 표시.
- **알림** — 일간/주간 요약을 이메일 또는 메신저(슬랙/카카오)로 발송.
- **주간 업무 요약** — 한 주 활동을 멤버·영역별로 자동 집계한 리포트.
- **구글 연동(미정)** — Drive 문서 연동 또는 Analytics 등, 필요 시 범위 확정.
- **네이버 블로그/일반 마케팅** — 공개 API가 빈약해 완전 자동화 어려움.
  RSS 수집 또는 빠른 입력 폼으로 보완.

## 9. 기술 아키텍처

### 9.1 스택

- **Next.js 16 (App Router) + TypeScript** — 웹앱
- **Supabase** — Postgres DB, 인증, 파일 스토리지
- **Tailwind CSS** — 스타일
- **Vercel** — 배포(GitHub 푸시 시 자동), 서버리스 함수
- 정기 작업(네이버 폴링) — GitHub Actions cron 권장

### 9.2 인증·권한

- 구글 OAuth (Supabase Auth).
- **이메일 화이트리스트** — 허가된 팀원 이메일만 로그인 허용.
- Row Level Security — 로그인한 팀원은 전체 데이터 읽기·쓰기(3인 내부 도구).
- 외부 연동 인증정보(GitHub·네이버)는 환경변수/시크릿으로만 관리.

### 9.3 배포

- GitHub 저장소 → Vercel 자동 배포.
- 환경변수는 Vercel 프로젝트 설정에 등록.
- Webhook 엔드포인트는 Vercel 서버리스 라우트로 노출.

## 12. 리스크와 미해결 질문

**리스크**

- 네이버 커머스 API는 판매자 계정의 **앱 등록**이 선행되어야 함 (사용자 작업).
- 네이버 블로그·일반 마케팅은 공개 API가 빈약 → 100% 자동화 불가, 보완 필요.
- Vercel Cron 빈도 제한 → 무료 유지하려면 GitHub Actions cron 사용.

**확정 필요 (확정해 주시면 기획에 반영)**

1. 팀원 3명의 역할 분담 — 각자 한 사업영역 전담인가, 함께 보는가?
   (담당자 기본값·기본 필터에 영향)
2. 로그인 허용 이메일 3개 — 화이트리스트에 넣을 주소.
3. 네이버 커머스 API 앱 등록을 진행할 수 있는 상태인가?
4. GitHub — 연동할 저장소 목록, 조직(Organization) 사용 여부.
5. Phase 4 "구글 연동" — 구체적으로 무엇을 원하는가? (미정으로 두어도 됨)
